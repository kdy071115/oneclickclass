# 원클릭 클래스 프론트엔드 테스트 시나리오

## 1. 목적과 기준

- 대상: 데스크톱 웹과 모바일 웹 프론트엔드 전체 기능
- 실행 URL: `BASE_URL` 환경 변수 사용. 로컬 기본값은 `http://localhost:5173`
- 테스트 사용자: 강의자, 공유 링크 방문자, 신청 완료 수강생
- 브라우저: Chromium 필수, Firefox·WebKit은 배포 전 교차 확인
- 반응형 기준: Desktop 1440x900, Tablet 1024x768, Mobile 390x844. 경계값은 1023/1024px, 767/768px도 확인
- 우선순위: `P0` 핵심 흐름, `P1` 주요 기능, `P2` 보조 기능
- 자동화: `E2E` Playwright, `CT` Vitest+RTL, `API` 계약 테스트, `Visual` 스냅샷, `Manual` 외부 연동 확인
- 판정 상태: `Automated`, `Partial`, `Planned`, `Manual`, `Backend-blocked` 중 하나로 관리

### 테스트 원칙

1. 화면 노출만 확인하지 않고 사용자 행동 이후 URL, 상태, 재조회 결과까지 검증한다.
2. 신청·결제·수강권은 서로 다른 상태로 검증하고 한 상태를 다른 상태로 추정하지 않는다.
3. 네트워크 요청은 횟수, payload, 응답 순서, 중복 호출 여부를 함께 확인한다.
4. 테스트 데이터는 시나리오별로 격리하고 실행 순서에 의존하지 않는다.
5. P0 실패, 콘솔 오류, `undefined` URL, 가로 스크롤은 배포 차단 사유로 처리한다.

## 2. 공통 사전 데이터

테스트 값은 UI 코드에 하드코딩하지 않고 fixture 또는 mock server에서 생성한다. 각 테스트는 생성한 ID를 사용하고 종료 후 데이터를 정리한다.

| Fixture | 조건 | 주요 검증 |
|---|---|---|
| `instructor` | 클래스 소유 강의자 | 생성·수정·공개·신청자 관리 권한 |
| `otherInstructor` | 소유권 없는 강의자 | 타인 클래스 403 및 액션 미노출 |
| `freeOpenCourse` | 무료·자동 승인·잔여 좌석 있음 | 신청 즉시 수강 가능 |
| `paidOpenCourse` | 유료·자동 승인·잔여 좌석 있음 | 결제 완료 후 수강 가능 |
| `approvalCourse` | 승인 필요 | 승인 전 수강 차단, 승인 후 조건 재평가 |
| `fullCourse` | `remainingSeats=0` | 신청 CTA 비활성 및 서버 충돌 처리 |
| `closedCourse` | 모집 마감 | 기존 수강생만 학습 가능 |
| `emptyCourse` | 공개 차시 없음 | 준비 중 상태와 되돌아갈 경로 제공 |
| `directVideoCourse` | 직접 업로드 영상 | native video 진도·완료·재시청 |
| `youtubeCourse` | YouTube 영상 | player event 기반 진도·완료·재시청 |
| `sequentialCourse` | 순차 학습 | 이전 차시 완료 전 다음 차시 잠금 |
| `assets` | JPG·PNG·WEBP 및 5MB 초과 파일 | 업로드 성공·실패·기존 이미지 보존 |

### 공통 상태 모델

| 영역 | 허용 상태 |
|---|---|
| 신청 | `APPLIED`, `APPROVED`, `REJECTED`, `CANCELLED` |
| 결제 | `NOT_REQUIRED`, `PENDING`, `PAID`, `FAILED`, `CANCELLED`, `REFUND_REQUESTED`, `REFUNDED` |
| 수강권 | `PENDING`, `AVAILABLE`, `SUSPENDED`, `COMPLETED`, `REVOKED` |
| 접근 판정 | 서버의 `canLearn`과 `accessReason`을 최종 기준으로 사용 |

프론트는 결제 완료만으로 수강 가능 여부를 추론하지 않는다. CTA, 안내 문구, 라우팅은 `canLearn`과 `accessReason` 조합으로 결정한다.

## 22. 테스트 데이터·실행 격리

- 테스트 이름 또는 `runId`를 클래스명에 포함해 병렬 실행 충돌을 방지한다.
- 고정 UUID, 특정 신청자 이름, 실행 순서에 기대지 않고 API 응답에서 받은 ID를 사용한다.
- 시간 의존 테스트는 시스템 시간을 고정하고 `Asia/Seoul`, UTC 변환, 자정·월말 경계를 포함한다.
- 결제·문자·YouTube는 PR에서 mock/fixture를 사용하고 staging에서 실제 sandbox 연동을 별도로 수행한다.
- 각 테스트는 필요한 데이터만 생성하고 `afterEach` 또는 전용 정리 API로 제거한다.
- 실패 시 URL, viewport, 네트워크 로그, 콘솔 오류, screenshot, trace를 남긴다. 인증정보와 개인정보는 마스킹한다.
- 시각 회귀는 동적 날짜·카운트다운·영상 프레임을 고정하거나 마스킹한 뒤 비교한다.

## 23. 자동화 실행 순서

1. PR 필수: lint, build, Vitest 컴포넌트 테스트
2. PR 필수 P0 E2E: 로그인, 클래스 생성, 무료 신청, 유료 결제 복귀, 신청자 승인, 수강 접근·진도·재시청
3. PR 필수 계약 테스트: 상태 조합, 결제 멱등성, 마지막 좌석 경쟁, 권한·소유권
4. PR 선택 P1 E2E: 알림, 정산, 설정, 필터·검색, 오프라인 복구
5. 배포 전: Chromium·Firefox·WebKit 핵심 흐름, 경계 viewport 시각 회귀, Axe 검사
6. staging: 문자 인증, 결제 sandbox, YouTube, SSE/QR, PDF 다운로드 실연동

## 24. 배포 판정 기준

### 배포 차단

- P0 실패 또는 핵심 흐름의 미실행
- uncaught exception, React Router 기본 오류 화면, 콘솔 error 발생
- 중복 신청·중복 결제·수강권 오발급·정원 초과
- 권한 없는 콘텐츠 접근 또는 개인정보·토큰 URL 노출
- 360~1920px에서 핵심 액션 가림, 가로 스크롤, 데스크톱의 강제 모바일 프레임
- 새로고침 전후 상태 불일치 또는 `undefined` 경로 이동

### 조건부 허용

- P2 실패는 담당자·수정 예정일·우회 방법을 기록한 경우에만 허용한다.
- 외부 연동 장애는 내부 상태를 오염시키지 않고 재시도 경로가 있을 때만 허용한다.

## 25. 현재 자동화 현황

현황은 기능 구현 여부와 자동화 여부를 분리해 관리한다. 기능이 화면에 존재하더라도 assertion과 fixture가 없으면 `Automated`로 표시하지 않는다.

| 묶음 | 현재 기준 | 다음 자동화 |
|---|---|---|
| 강의자 핵심 흐름 | 클래스 생성·수정·상세·커리큘럼 흐름 존재 | `CREATE-001~011`, 소유권·중복 저장 보강 |
| 공개 신청 | 공유 페이지와 신청 UI 존재 | `AUTH-011~015`, `ENROLL-001~012` |
| 학습 | 직접 영상·YouTube·진도 UI 존재 | `LEARN-004~014`, 공급자별 fixture 분리 |
| 신청자 관리 | 목록·상세·상태 액션 존재 | 상태 3축 필터, 새로고침 일관성 |
| 운영 기능 | 알림·출석·설문·수료증·정산 UI 존재 | 실제 API 계약과 빈 상태·오류 상태 보강 |

백엔드가 아직 제공하지 않는 멱등키, 좌석 홀드, 결제 콜백, 수강권 회수 검증은 `Backend-blocked`로 표시하되 프론트 mock 시나리오는 먼저 유지한다.
