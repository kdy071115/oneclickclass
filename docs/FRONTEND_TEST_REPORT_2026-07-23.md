# 프론트엔드 테스트 실행 보고서

- 실행일: 2026-07-23
- 기준 문서: [프론트 테스트 시나리오](./FRONTEND_TEST_SCENARIOS.md)
- 대상 브랜치/작업 트리: 현재 로컬 작업 트리
- 실행 환경: macOS, Chromium, Vite production preview

## 1. 결론

현재 구현된 자동화와 프론트 mock 범위는 정상 동작한다. 다만 기준 문서의 172개 시나리오 전체를 배포 통과로 판정할 수는 없다.

- 정적 검사, production build, Vitest 36건은 모두 통과했다.
- Playwright 36건 중 적용 viewport에 해당하는 26건이 통과했고, 10건은 테스트에 명시된 조건에 따라 skip됐다.
- 40개 라우트를 4개 viewport에서 총 160회 탐색했으며 런타임 예외, `/undefined`, `/null`, 문서 가로 넘침은 발견되지 않았다.
- `AUTH-009`는 실패다. 제거 대상인 `/guest` 라우트와 `GuestPage`가 아직 노출된다.
- 실결제, 문자, 서버 권한, 좌석 동시성, 결제 콜백 멱등성은 실제 백엔드나 sandbox 없이 통과 판정할 수 없다.

## 3. E2E에서 확인한 흐름

- 보호 라우트 로그인과 대시보드 복귀
- 강의 생성 단계 검증, 일정 선택, 커리큘럼 연결
- YouTube URL 검증, 차시 공개, 신청 페이지 공개, 수강생 강의실 입장
- 공개 신청 필수값, 휴대전화 인증번호, 무료·유료 CTA
- 강의자 상세 탭과 신청자·출석·설문·시험·수료증·설정 이동
- 신청자 상세, 결제 확인, 메시지, 긴 신청 답변 레이아웃
- 알림 팝오버 목적지와 모두 읽음
- 출석 QR 갱신, 설문/시험 생성과 결과, 수료증 발급 설정
- 커리큘럼 섹션·차시 추가 및 수정값 유지
- 모바일 신청·학습·수료증과 1024/1280px 강의자 화면 가로 넘침

## 4. 상태·학습 단위 테스트에서 확인한 계약

- 무료 신청은 `NOT_REQUIRED` 결제와 즉시 접근 상태 사용
- 유료 신청은 결제 전 `PENDING`, 결제 완료 후 `PAID`로 전환
- 최초 신청과 새 기기 이어보기에서 인증번호 필요
- 공개 페이지와 학습실이 강의자 커리큘럼을 동일하게 사용
- 공개 차시가 없는 신규 강의에 데모 차시를 주입하지 않음
- heartbeat 이후 학습 위치 저장과 완료 상태 유지
- 선행 차시 완료 후 순차 학습 잠금 해제
- 직접 영상·YouTube 등 콘텐츠 공급자와 잘못된 URL 구분
- 클래스별 신청자·설문·시험·출석·수료증 데이터 격리
- 강의 공개 전 비공개 상태와 공개 후 상태 변경

## 5. 전체 라우트 스모크

다음 화면을 `360x800`, `768x900`, `1024x768`, `1440x900`에서 확인했다.

- 강의자: 대시보드, 클래스 목록·상세·생성·커리큘럼·미리보기, 신청자, 출석, QR, 설문, 시험, 수료증, 정산, 알림, 설정, 결제, 고객센터
- 수강생: 공개 신청, 학습실, 설문 응시·완료, 시험 응시·결과, 마이·수료증
- 인증·예외: 로그인, 회원가입, 기존 비회원 조회, 존재하지 않는 경로

검사항목은 navigation 실패, page error, React Router 기본 오류 화면, `removeChild` 오류, `/undefined`·`/null` URL, 문서 폭 초과다. 해당 기술 오류는 발견되지 않았다.

화면이 열렸다는 사실은 각 버튼의 업무 규칙이 맞다는 뜻이 아니다. 라우트 스모크는 E2E assertion을 대체하지 않는다.

## 6. 발견 사항

### FAIL-001 기존 비회원 조회가 남아 있음

- 연결 시나리오: `AUTH-009`
- 실제: `/guest`가 `GuestPage`를 렌더링하며 휴대전화 기반 조회 UI를 제공한다.
- 기대: 기존 조회 화면을 제거하고 공유 링크 신청 및 새 기기 이어보기 인증으로 통일한다.
- 영향: 신규 신청과 기존 수강권 복구의 진입점이 중복되어 사용자가 어느 흐름을 사용해야 하는지 혼란스럽다.
- 제안: `/guest` 라우트와 메뉴 진입점을 제거하고 직접 접근은 안내 가능한 공유 링크 또는 404로 처리한다.

### WARN-001 production chunk 크기

- production JavaScript가 약 587kB이며 Vite의 500kB 경고를 발생시킨다.
- 실제 LCP·INP와 캐시 결과를 측정한 뒤 강의자 운영 화면과 수강생 화면의 route-level lazy loading을 검토한다.

### GAP-001 브라우저·접근성 자동화

- 현재 E2E project는 Desktop Chrome과 iPhone 13 WebKit 설정이 아니라 Chromium 기반 mobile emulation이다.
- Firefox, 실제 WebKit, Axe, 200% 확대, 키보드 전수 이동은 아직 자동 검증되지 않았다.

## 9. 배포 판정

현재 판정은 **조건부 미통과**다.

- 프론트 구현의 현재 자동화 범위는 통과했다.
- `AUTH-009` 제품 불일치가 남아 있다.
- 결제·좌석·수강권·권한의 서버 계약 테스트가 완료되지 않아 전체 서비스 배포 통과로 선언할 수 없다.
